Clean Architecture in Go: A Practical Guide with Real Project Structure

📅 May 28, 2026 📖 5 min read ✏️ Arif Kurniawan
🗄️

When I first started building Go services, I followed the classic pattern: handlers/, models/, utils/. It worked for a while, but as the project grew, every change felt risky. Tests became brittle. Business logic leaked into handlers.

That's when I adopted Clean Architecture — and it changed how I write Go entirely. Here's the structure I now use in every production service.

The Directory Layout

my-service/
├── cmd/
│   └── server/          # Entry point — wire everything together
├── internal/
│   ├── domain/          # Entities + repository interfaces
│   ├── usecase/         # Business logic — one file per use case
│   ├── repository/      # Implementations (PostgreSQL, in-memory, etc.)
│   ├── handler/         # HTTP / gRPC handlers
│   └── config/          # Configuration loading
├── pkg/
│   └── middleware/      # Shared middleware
└── go.mod

Why This Works

The key insight: dependencies point inward. The domain layer knows nothing about HTTP, databases, or frameworks. The usecase layer depends on interfaces defined in domain, not concrete implementations.

"Clean Architecture is not about folder structure. It's about the dependency rule — source code dependencies must point inward, toward the domain."

Real Example: User Registration

Here's what it looks like in practice. First, the domain interface:

// internal/domain/user.go
package domain

import "context"

type User struct {
    ID        string
    Email     string
    Password  string  // hashed
    CreatedAt int64
}

type UserRepository interface {
    Save(ctx context.Context, user *User) error
    FindByEmail(ctx context.Context, email string) (*User, error)
}

Then the use case — notice it only knows about UserRepository, not SQL:

// internal/usecase/user.go
package usecase

type RegisterUserInput struct {
    Email    string
    Password string
}

func (uc *UserUsecase) Register(ctx context.Context, input RegisterUserInput) (*domain.User, error) {
    existing, _ := uc.repo.FindByEmail(ctx, input.Email)
    if existing != nil {
        return nil, ErrEmailAlreadyExists
    }

    hashed, _ := bcrypt.GenerateFromPassword([]byte(input.Password), 12)
    user := &domain.User{
        ID:        uuid.NewString(),
        Email:     input.Email,
        Password:  string(hashed),
        CreatedAt: time.Now().Unix(),
    }

    if err := uc.repo.Save(ctx, user); err != nil {
        return nil, fmt.Errorf("save user: %w", err)
    }

    return user, nil
}

Testing Becomes Trivial

Because business logic depends only on interfaces, you test with simple mocks — no database needed:

func TestRegisterUser(t *testing.T) {
    mockRepo := new(MockUserRepository)
    mockRepo.On("FindByEmail", mock.Anything, "test@email.com").Return(nil, nil)
    mockRepo.On("Save", mock.Anything, mock.Anything).Return(nil)

    uc := NewUserUsecase(mockRepo)
    user, err := uc.Register(context.Background(), RegisterUserInput{
        Email:    "test@email.com",
        Password: "securePass123!",
    })

    assert.NoError(t, err)
    assert.Equal(t, "test@email.com", user.Email)
    mockRepo.AssertExpectations(t)
}

What I Learned Along the Way

Clean Architecture isn't a silver bullet. For tiny services (2-3 endpoints), the overhead isn't worth it. But for any service expected to live more than 6 months, the investment pays for itself many times over.

At KB Finansia Multi Finance, rewriting a legacy car platform into Clean Architecture cut our bug rate by ~60% and made onboarding new engineers take days instead of weeks.

Common Pitfalls to Avoid

  • Over-abstracting too early — Don't create interfaces until you have a second implementation. Start concrete, extract interfaces when needed.
  • Leaking framework details into use cases — No gin.Context, echo.Context, or HTTP concepts in internal/usecase/.
  • Giant use case files — If a file has 10+ methods, it's probably doing too much. Split by domain aggregate.
  • Skipping domain types — Raw string parameters everywhere means you lose type safety. Define Email, PhoneNumber value objects.
