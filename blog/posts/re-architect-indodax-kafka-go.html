From PHP to Kafka & Go: How We Re-Architected Indodax's Core Exchange Engine

📅 Jun 6, 2026 📖 8 min read ✏️ Arif Kurniawan
🏗️

When I joined Indodax, the core exchange engine was running on a legacy PHP monolith. It worked — but every deployment was nerve-wracking, a single bug could cascade across the entire system, and scaling meant throwing more servers at it.

Over 18 months, we migrated it to a Kafka-based event-driven architecture with Go services. This is the story of how we did it — and the hard lessons we learned along the way.

The Starting Point: A PHP Monolith

The original system processed orders, matched trades, and updated balances all in a single PHP process. The flow looked roughly like:

HTTP Request → PHP Router → Order Handler → MySQL (orders + balances) → Response

The problems were obvious:

  • No separation of concerns — order matching, balance updates, and notifications all lived in the same code path
  • Database contention — every trade meant row-level locks on both orders and balances tables
  • Hard to test — you couldn't test order matching without the full PHP stack and database
  • Scaling meant vertical — you couldn't independently scale order ingestion vs matching vs reporting

Target Architecture: Event-Driven with Kafka

We chose event-driven architecture with Kafka because it offered:

  • Decoupled services — each component scales independently
  • Durable event log — replayability for debugging and reconciliation
  • Exactly-once semantics — critical for financial transactions
  • Async processing — order ingestion doesn't block matching

Migration Strategy: Strangler Fig

We didn't do a big-bang rewrite. Instead, we used the Strangler Fig pattern:

  1. Phase 1 — Parallel run: New Go services processed events alongside the PHP system. We compared outputs to validate correctness.
  2. Phase 2 — Read path: Queries (order history, balance checks) moved to Go first since they're read-only.
  3. Phase 3 — Write path: Order placement migrated to Go with the PHP system as read-only fallback.
  4. Phase 4 — Decommission: Once all traffic ran on the new system, we turned off the old PHP stack.
"The key to a successful migration is not how fast you can build the new system — it's how safely you can run both in parallel while validating every output."

The Kafka Topic Design

Topic design was the most debated part of the architecture. We settled on this:

orders.pending      → Partitioned by order_id
orders.matched      → Partitioned by user_id (for balance updates)
orders.cancelled    → Compacted topic (latest state per order)
balances.events     → Single partition (strict ordering for balance accuracy)
trades.executed     → Partitioned by trading_pair

The single partition for balances was controversial — it limits throughput. But for a cryptocurrency exchange, balance accuracy is non-negotiable. We accepted the throughput ceiling in exchange for deterministic ordering.

What Went Wrong (and What We Fixed)

1. Consumer Lag During High Volatility

When Bitcoin price spiked, order volume jumped 10x. Consumers couldn't keep up. We fixed this by:

  • Increasing partition count per topic (requires careful planning — can't change without re-partitioning)
  • Adding consumer autoscaling based on consumer lag metrics
  • Caching hot trading pairs (BTC/IDR, ETH/IDR) in Redis to reduce database load

2. Exactly-Once is Hard

Kafka's exactly-once semantics require the producer, broker, and consumer all to participate. We hit edge cases with idempotent producers and transactional consumers. The lesson: design your system to tolerate duplicates at the application level rather than relying solely on Kafka's exactly-once guarantees.

3. State Reconstruction from Events

When a consumer crashes and replays events, it needs to reconstruct its in-memory state. We implemented a snapshot mechanism that periodically persisted state and used Kafka offsets to know where to resume replay.

Results

After the migration:

  • 3x improvement in order processing throughput
  • Zero-downtime deployments — each service deploys independently
  • 95% reduction in database lock contention
  • Full replay capability — we can rebuild any user's balance from the event log

Key Takeaways

If you're planning a similar migration:

  1. Don't rewrite everything at once — the Strangler Fig pattern is your friend
  2. Invest in observability early — Kafka consumer lag, error rates, and latency metrics are essential
  3. Test with real traffic patterns — simulate high volatility scenarios before going live
  4. Plan for exactly-once at the application level — idempotent handlers prevent duplicate processing
  5. Go is worth it for latency-sensitive paths — Go's goroutines and channels map naturally to event-driven patterns

The migration wasn't easy, but it transformed how we build and deploy at Indodax. Today, adding a new feature means shipping a new Kafka consumer — not touching a monolithic PHP codebase.
